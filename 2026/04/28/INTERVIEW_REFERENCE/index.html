1. 项目概述

1.1 一句话描述

一个面向培训机构的 员工与部门管理后台 API 系统，支持员工 CRUD、JWT 认证、操作日志 AOP 自动记录、文件上传至阿里云 OSS、以及数据统计报表。

1.2 业务背景

培训机构（类似「Tilias」）需要管理内部员工——包括班主任、讲师、学工主管、教研主管、咨询师等角色。管理员需要：

  • 对部门和员工进行增删改查
  • 记录每位员工的工作经历（简历）
  • 查看员工职位分布、性别比例等统计图表
  • 追踪所有敏感操作（谁、什么时间、做了什么、耗时多久）
  • 上传员工照片等文件到云端

1.3 项目定位

  • 类型：RESTful API 后端服务（前后端分离架构）
  • 前端：独立的 Vue/React 项目（本项目不包含）
  • 数据库：MySQL 8.x
  • 部署方式：Jar 包独立运行（Spring Boot 内嵌 Tomcat）

7. 关键设计决策

7.1 为什么用 MyBatis 而不是 JPA？

  • SQL 可控性强——复杂统计查询（职位分布、性别统计）手写 SQL 更直观
  • 学习价值高——理解 ORM 底层原理
  • 混合模式——简单 CRUD 用注解，复杂查询用 XML

7.2 为什么用 ThreadLocal 传递用户信息？

  • 避免在每个方法签名中传递 userId
  • 请求处理是单线程的，天然隔离
  • 关键：必须 finally remove()，否则线程池复用会导致串用户

7.3 为什么 JWT 密钥不能随机生成？

原来的代码 Keys.secretKeyFor(HS256) 每次类加载时生成新密钥 → 应用重启后所有已发 token 失效。改为固定密钥（实际项目应从配置中心/环境变量读取）。

7.4 MyBatis statement ID 的坑

MyBatis XML 的 <select id="..."> 严格区分大小写，必须与 Mapper 接口方法名逐字一致。例如：

  • 接口：genderListMaps() → XML 必须：id="genderListMaps"（不能写成 GenderlistMaps）
  • 接口：deleteByIds() → XML 必须：id="deleteByIds"（不能写成 deleteById）

否则抛出 BindingException: Invalid bound statement。

  • 接口：genderListMaps（小写 g）

9. 面试常见问题与回答

Q1: 介绍一下你这个项目？

这是一个培训机构员工管理系统后端，基于 Spring Boot 3 + MyBatis + JWT。核心功能包括部门和员工的 CRUD、JWT 登录认证、操作日志 AOP 自动记录、文件上传到阿里云 OSS、以及员工职位和性别统计报表。

技术栈方面用到了：Spring AOP 做切面日志、PageHelper 做物理分页、ThreadLocal 做请求级用户信息传递、JWT + 拦截器做认证，事务管理用了 REQUIRES_NEW 保证审计日志独立落库。

这是我大二独立完成的，目前代码量大约 30 个 Java 文件。

Q2: JWT 认证是怎么做的？

生成：用户 POST /login 提交用户名密码 → EmpMapper.login() 查库 → 如果查到用户，用 JwtUtils.generateToken() 生成 JWT，载荷中放 id 和 username，HS256 签名，有效期 12 小时。

校验：所有请求（除 /login）经过 TokenInterceptor → 从 Header 取 token → JwtUtils.parseToken() 校验签名和过期时间 → 提取用户 id → 存入 CurrentHolder（ThreadLocal）。

清理：请求结束后 afterCompletion() 中调用 CurrentHolder.remove() 防止内存泄漏——因为 Tomcat 线程池复用，不清理会导致线程下次被复用时串用户。

Q3: 为什么用 ThreadLocal？有什么风险？

Spring 的 Controller 默认是单例的，多个请求共享同一个 Controller 实例。如果我在 Controller 里用成员变量存用户信息，多个请求会互相覆盖。

ThreadLocal 给每个线程一个独立副本——一个 HTTP 请求从进来到离开始终在同一个线程中处理（Tomcat 线程模型），所以 ThreadLocal 天然隔离不同用户的请求。

风险：线程池复用导致内存泄漏和串用户。必须在 finally 或拦截器的 afterCompletion 中 remove()。

这里还有一个细节：/login 路径被 WebConfig 排除了拦截器，所以不会走到 ThreadLocal 的 set，但 TokenInterceptor 只做 set，所以无影响。

Q4: AOP 操作日志怎么实现的？

我写了一个自定义注解 @LogOperation，标注在 Controller 方法上。

LogAspect 切面类用了 @Aspect + @Component，通过 @Around("@annotation(LogOperation)") 拦截所有标注了该注解的方法。

在环绕通知里：先记开始时间 → joinPoint.proceed() 执行目标方法 → 计算耗时 → 从 ThreadLocal 拿操作人 ID → 构建 OperateLog 对象（类名、方法名、参数、返回值、耗时）→ OperateLogMapper.insert() 写入数据库。

这样做的好处是声明式、零侵入——以后想记录哪个操作，加个注解就行，不用改业务代码。

Q5: 事务是怎么管理的？遇到什么特殊情况？

Spring 的声明式事务 @Transactional。员工新增 add() 方法标注了 @Transactional(rollbackFor = Exception.class)。

特殊情况：操作日志的写入。我在 EmpLogServiceImpl.insertLog() 上用了 @Transactional(propagation = Propagation.REQUIRES_NEW)。

原因：如果新增员工的主事务回滚了（比如工作经历插入失败），操作日志不能跟着回滚——审计日志必须保留。REQUIRES_NEW 会挂起当前事务，新开一个独立事务，日志写入后立即提交，不受主事务回滚影响。

同时用了 finally 块保证即使主事务抛异常，日志代码也一定执行。

Q6: 分页怎么做的？为什么用物理分页？

用了 PageHelper 插件。PageHelper.startPage(page, pageSize) 后紧跟着 MyBatis 查询，PageHelper 通过 MyBatis 拦截器自动在 SQL 前加 SELECT count(0) 查总数，然后在原 SQL 上加 LIMIT offset, size。

物理分页 vs 逻辑分页：物理分页只查需要的数据量，百万级数据也不怕；逻辑分页（把全部数据查出来再内存截取）数据量大直接 OOM。

Q7: 文件上传怎么做的？安全方面考虑了哪些？

用了阿里云 OSS SDK。AliyunOSSOperator.upload() 接收文件字节数组和原始文件名：

  1. 从环境变量读取 AK/SK（不是硬编码）
  2. 按日期 yyyy/MM 分目录
  3. UUID 生成唯一文件名防覆盖
  4. V4 签名算法
  5. finally 中 shutdown OSS 客户端防连接泄漏

安全方面：凭证走环境变量、V4 签名、UUID 防遍历。

Q8: MyBatis 怎么用的？注解和 XML 有什么区别？

我采用了混合模式。简单 SQL（部门 CRUD、插入日志）用 @Insert / @Select / @Delete 注解直接写在 Mapper 接口上。

复杂 SQL 用 XML 写——比如员工列表查询有 4 个可选过滤条件，XML 里用 <if> 动态拼接 WHERE 子句；员工详情查询需要 JOIN 两张表（dept + emp_expr）用 <resultMap> + <collection> 做一对多映射。

我在这个过程中踩过坑：XML 的 statement id 必须和接口方法名严格一致（包括大小写），否则 MyBatis 找不到 SQL 会抛 BindingException。

Q9: 如果有 1000 个人同时访问，你的系统能撑住吗？

目前的单机部署，Tomcat 默认线程池 200 线程。1000 并发的话：

  • 数据库连接池是瓶颈（需检查 HikariCP 配置，默认 10 连接远不够）
  • 需要加连接池大小、加索引、上 Redis 缓存热点数据
  • 文件上传是同步的，大文件会阻塞线程
  • 如果要扛高并发，需要加 Nginx 反向代理 + 多实例 + 读写分离

但这对于培训机构内部管理场景来说，并发不会超过几十人，当前架构完全够用。

Q10: 你在这个项目中最大的收获是什么？

第一是理解了拦截器 + ThreadLocal 这种请求级上下文传递模式，很多框架（Spring Security、事务管理器）底层都是这么做的。

第二是 AOP 的实战应用——不只是学了个概念，而是真的用它解决了操作日志这个横切关注点。

第三是踩了一些真实的坑：MyBatis 的 statement ID 大小写问题、JWT 密钥随机生成导致 token 失效、事务传播级别理解不到位导致日志回滚——这些都是在调试中发现的，和只看书完全不同的体验。
