软件工程笔记：初探形式化方法与面向对象建模 (UML)

在近期的软件工程课程中，老师布置了两个核心的学习方向：一是理解形式化方法（Formal Methods），二是阅读经典著作**《大象——Thinking in UML》**。

这篇博客将作为我学习这两个重要概念的起点，记录我的初步认知与后续的学习计划。

一、 什么是“形式化方法”？

在日常编程中，我们习惯用自然语言写需求，用代码实现逻辑，然后通过测试来找 Bug。但测试只能证明 Bug 的存在，不能证明 Bug 不存在。

形式化方法（Formal Methods） 则提供了一种降维打击的思路：
它是一种基于严格数学基础的软件开发技术。它使用离散数学、逻辑学的方法，像做数学证明题一样，去推导和验证软件系统的正确性。

1. 为什么需要形式化方法？

当代码出错的代价极其惨痛时（例如：航空航天系统、高铁信号控制、核电站、医疗仪器），传统的“修修补补”是不可接受的。形式化方法可以在写代码之前，从逻辑根源上证明系统设计是没有漏洞的。

2. 它的局限性在哪？

  • 学习门槛极高：需要开发人员具备极强的离散数学和数理逻辑功底。
  • 开发成本巨大：耗时极长，通常只用于系统中最核心、最致命的部分，一般互联网公司的 CRUD 业务几乎用不到。

💡 个人感悟：虽然我们日常开发很少直接用到它，但它背后那种“严谨求证、状态穷举”的思维方式，对写出健壮的业务代码极具启发意义。

二、 《大象：Thinking in UML》阅读指南

老师推荐的第二部分是阅读《大象——Thinking in UML》。这本书在国产软件工程书籍中地位极高。

1. 为什么叫“大象”？

书名取自“盲人摸象”的典故。在复杂的软件开发中，前端看到的是界面，后端看到的是接口，DBA 看到的是表结构——大家都是“盲人”。而 UML（统一建模语言） 就是那套帮助我们完整描绘出“大象”全貌的工具和沟通语言。

2. 这本书的核心价值

市面上很多书都在教“UML 怎么画”，比如箭头怎么画、方框代表什么。但《大象》重点讲的是 OOAD（面向对象分析与设计） 的思想。它教我们如何从一团乱麻的用户需求中，抽象出业务模型。

3. UML 核心图例学习规划

接下来我将重点攻克以下几种 UML 图，并在后续博客中更新实践心得：

  • 用例图 (Use Case Diagram)：明确系统边界，“谁”在用系统做什么？
  • 类图 (Class Diagram)：系统的骨架，面向对象设计的核心体现。
  • 状态图 (State Machine Diagram)：描述复杂对象在不同条件下的状态流转（比如订单状态机）。
  • 时序图 (Sequence Diagram)：梳理系统各个组件或微服务之间的交互顺序。

三、 总结与后续 Action

软件开发不仅仅是敲代码（Coding），更重要的是前期的分析与设计（Design）。形式化方法代表了设计的“极致严谨”，而 UML 则是设计的“工程化表达”。

下一步计划：我将结合 Java 后端的实际业务场景（如：登录授权或支付流程），尝试用 UML 画出完整的业务模型，并发布在博客上，敬请期待！
