面向对象设计：开闭原则 (OCP) 与图表显示系统重构

在面向对象设计中，开闭原则（Open-Closed Principle, OCP） 是软件实体设计的终极目标。它的核心思想是：对扩展开放，对修改关闭。 换句话说，当我们需要给系统增加新功能时，应该尽量通过增加新代码来实现，而不是去修改现有的、已经运行良好的代码。

今天课堂上的练习题完美展示了违反开闭原则的痛点，本文将记录如何运用多态和抽象机制对其进行重构。

一、 需求背景与原始设计缺陷

1. 案例说明

Sunny 软件公司开发的 CRM 系统需要显示各种类型的图表（如饼状图、柱状图等）。
在原始设计中，有一个 ChartDisplay 类，它包含一个 display(String type) 方法。该类直接依赖于具体的 PieChart 和 BarChart 类。

2. 为什么它违反了开闭原则？

致命痛点：如果明天产品经理要求系统中增加一种“折线图 (LineChart)”，我们就必须修改 ChartDisplay 类的源代码（加上一个 else if）。每次增加新图表都要修改基础展示类，这极容易引入新的 Bug，系统完全是“对修改开放”的。

二、 重构方案：引入抽象层

要实现开闭原则，最常用的手段就是面向抽象编程。我们需要在 ChartDisplay 和具体的图表类之间，增加一个抽象层。

1. 定义抽象图表接口 (AbstractChart)

将所有图表的共性抽取出来，定义一个抽象类或接口。

2. 具体图表实现抽象接口

让现有的饼图、柱状图（以及未来可能新增的任何图表）去实现这个接口。

3. 重构 ChartDisplay 类 (彻底解耦)

ChartDisplay 不再依赖具体的图表类，而是依赖抽象接口 AbstractChart。利用 Java 的多态特性，动态决定调用哪个图表的 display 方法。

三、 重构后的优势分析

经过这样的重构，我们再来看看如果现在需要增加“折线图 (LineChart)”该怎么办：

  1. 新增代码：只需新建一个 LineChart 类实现 AbstractChart 接口。
  2. 无需修改：完全不需要改动 ChartDisplay 类的任何一行代码！

我们在不修改原有系统核心代码的前提下，完美扩展了系统功能。这就实现了对扩展开放（可以随便加新图表），对修改关闭（原有的显示类不需要动），大大提高了系统的可维护性和稳定性。
