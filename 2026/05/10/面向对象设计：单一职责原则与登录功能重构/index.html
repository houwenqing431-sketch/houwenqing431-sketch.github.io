面向对象设计：单一职责原则 (SRP) 与登录功能重构

在面向对象设计中，单一职责原则（Single Responsibility Principle, SRP） 是最重要的原则之一。它的核心思想是：一个类应该只有一个引起它变化的原因。

今天在课堂上遇到了一个非常经典的重构案例：一个基于 Java C/S 架构的登录功能实现。本文将记录如何运用单一职责原则对其进行解耦与重构。

一、 需求背景与问题分析

1. 原始设计（反面教材）

系统原先通过一个 Login 类来实现整个登录流程，其类图方法如下：

  • init() : void
  • display() : void
  • validate() : void
  • getConnection() : Connection
  • findUser(String userName, String userPassword) : boolean
  • main(String args[]) : void

2. 为什么它违反了单一职责原则？

分析这个 Login 类，我们会发现它承担了三个完全不同的职责：

  1. UI 界面展示：init() 和 display() 负责界面的初始化和绘制。
  2. 数据库连接管理：getConnection() 负责获取数据库连接。
  3. 业务逻辑与数据访问：findUser() 负责去数据库校验账号密码，validate() 可能负责前端或后端的校验。

带来的痛点：
如果有一天我们需要把界面从 C/S 架构（比如 Java Swing）改成 B/S 架构（Web 页面），我们需要修改这个类；如果我们需要把数据库从 MySQL 换成 Oracle，我们还是需要修改这个类。代码高度耦合，牵一发而动全身。

二、 重构方案：职责分离

根据单一职责原则，我们需要将这个“全能”的类拆分成三个功能单一的类：一个管界面，一个管数据库连接，一个管业务查询。

1. 界面类 (LoginForm)

专门负责与用户的交互，收集用户输入的数据，并展示结果。

2. 数据库工具类 (DBUtil)

专门负责底层数据库的连接与资源管理。任何需要操作数据库的地方都向它获取连接。

3. 数据访问/业务逻辑类 (UserDAO)

（DAO = Data Access Object）。专门负责处理与用户相关的业务逻辑，与具体的 UI 界面彻底脱钩。

三、 总结

重构后，各个类的职责变得极其清晰：

  • 如果以后 UI 界面要重构，只需要修改 LoginForm。
  • 如果以后数据库连接池换了（比如换成 HikariCP），只需要修改 DBUtil。
  • 如果以后密码校验逻辑加入了 MD5 加密，只需要修改 UserDAO。

高内聚，低耦合——这就是单一职责原则为我们代码带来的最大价值。
